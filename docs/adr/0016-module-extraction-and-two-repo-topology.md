# ADR-0016: Module Extraction and Two-Repo Topology

## Status
Accepted

## Date
2026-07-09

## Context

This repo is a flat, single-root-module OpenTofu project running one production Talos node on an OVH dedicated server. Three goals now require restructuring it (strategy review, 2026-07-08/09):

1. **Reusable module**: a provider-agnostic Talos cluster module (`talos-core` + per-provider node adapters) intended for registry publication. A prior-art survey (2026-07-08) found no existing equivalent: the mature Talos modules (hcloud-talos/terraform-hcloud-talos ~345★, hcloud-k8s/terraform-hcloud-kubernetes ~684★, sergelogvinov/terraform-talos ~205★) all hard-couple Talos config generation to one provider, and no module exists at all for Talos on OVH dedicated servers.
2. **Multi-node readiness** (planned ADR-0017, tracked in #310): the module API must express N nodes, roles, and adapters even while production stays single-node.
3. **DR-first operations** (planned ADR-0018, tracked in #311): independent of this ADR, but the module boundary decides where storage/backup live.

Hard constraints the topology must respect:

- **This repo is public** with live GitOps (ADR-0014: dflook plan/apply/drift), `production` environment secrets, CODEOWNERS-gated merges, signed releases with SBOM. Its operational hardening (state lineage, secrets, rulesets, crons, tailnet ACL trust) took months to accumulate.
- **State fragility history**: #268 required manual state surgery after a trigger-identity change cascaded toward a full server reinstall; the standing rule is that reinstall-trigger identity never changes outside a planned reinstall window. #282 (sha256-wrapping identifier-bearing trigger entries) was accepted-and-deferred — the issue is closed and the reminder is encoded as a NOTE in `main.tf` — and executes only at the next planned reinstall.
- **Two production-state write origins already exist**: upgrade-mode applies run `talosctl upgrade` via local-exec and the GitOps runner has no talosctl, so production state is written both by consumer-repo CI and by the operator's workstation, under one object-storage cloud user.
- **Public-log identifier exposure**: GitHub echoes step `env:` blocks and masks secrets but **not** variables — infrastructure identifiers leaked once (#300) and were moved to environment secrets.
- **OVH S3 specifics**: per-object ownership means all state writers must share one cloud user, and each cloud user is limited to two S3 keys — a new state-writer identity cannot be added casually.
- **Registry requirements**: both registries index public GitHub repos named `terraform-<system>-<name>` with semver tags; the OpenTofu registry takes submissions via a GitHub issue form and then auto-indexes new tags. The current repo name does not satisfy the pattern.
- **No HashiCorp/IBM tooling** (ADR-0014).

## Considered Options

| Option | Reuse/registry | State & secrets migration | e2e credential isolation | Verdict |
|---|---|---|---|---|
| **New public module repo; this repo becomes the consumer (zero-diff refactor)** | Full | **None** — state lineage, writer identity, secrets, rulesets, crons all stay | Full (module repo never sees prod creds) | **Selected** |
| New private `xd-infra` consumer repo; this repo archived | Full | Re-provision ~12 secrets + variables, rulesets, crons; migrate state to a new backend identity (trips OVH per-object ownership + 2-key limit); transition window with two repos able to write prod state | Full | Rejected — rebuilds months of hardening to gain nothing the selected option lacks |
| Internal `modules/` split only, no extraction | None (registry naming unsatisfiable; no community reuse) | None | n/a | Rejected as end state; acceptable as a development stepping stone |
| Monorepo: module + consumer in one repo | Leaky — consumers fetch the live consumer's backend/GitOps files in every tag archive; a published library housing a production consumer is structurally wrong | None | **Impossible — prod creds and fork-facing e2e share one trust domain (the decisive disqualifier)** | Rejected |
| Monorepo dev + subtree-split to a clean published mirror | Full | None | Full (mirror carries e2e; source keeps prod) | Rejected — a sync pipeline to maintain for a solo team, and the source monorepo still co-locates prod creds with fork-facing development; the selected option achieves the same isolation without split-mirror machinery |

## Decision

**Extract the module into a new public repo; refactor this repo into its consumer behind a zero-diff gate.**

1. **Module repo**: `xd-ventures/terraform-talos-cluster` (Apache-2.0), registry address `xd-ventures/cluster/talos`. The registry `<system>` slug is `talos` — the `siderolabs/talos` config-plane provider and the module's domain — not an infrastructure provider, because the module is provider-agnostic; the infra provider is chosen by which node adapter the consumer wires in. Layout: the **repo root hosts a thin composition module** (this is what the registry address resolves to), with the building blocks under `modules/` — `talos-core` (provider-agnostic: machine secrets, per-role/per-node machine configs, bootstrap, Cilium, Tailscale identity, role-aware firewall), `node-ovh-baremetal` (extracted BYOI machinery, kept adapter-internal), `node-ovh-publiccloud` (new, OpenStack) — individually addressable as `xd-ventures/cluster/talos//modules/<name>` (git fallback: `git::https://github.com/xd-ventures/terraform-talos-cluster.git//modules/<name>?ref=vX.Y.Z`). Adapters implement a uniform contract (node IPs, a provisioned handle, a per-node replace trigger) so new providers never change `talos-core`. The module declares `required_providers` only; all `provider {}` blocks live in consumer roots (#321–#325).
   **License continuity**: LICENSE copied verbatim; the license-eye SPDX-header check is replicated so every extracted file keeps its `Apache-2.0` header and copyright line; the module README carries a provenance note (extracted from `xd-ventures/tf-xd-venture-talos01`; the prior-art modules in References were surveyed for design, not copied). This repo has no NOTICE file, so Apache-2.0 §4(d) imposes no NOTICE carry-forward — none is added. The module repo starts with fresh git history plus the provenance note (the source repo's history was already scrubbed once; re-filtering it buys nothing).

2. **Consumer**: this repo. It keeps its state lineage, backend, S3 writer identity, environment secrets, rulesets, and cron workflows unchanged. No new company repo is created until a second cluster actually exists.

3. **Zero-diff cutover gate** (#326): the consumer switches to `module` blocks in one PR containing complete `moved` blocks and only value-preserving changes — resource-address relocation plus the grandfathered variable renames (whose resolved values are unchanged, so rendered machine config stays byte-identical). The pre-merge gate is a **moves-only plan**: every operation a `moved` block, `0 to add, 0 to change, 0 to destroy` (OpenTofu never prints "No changes" while moves are pending; `-detailed-exitcode` returns 2). The post-merge apply consumes the moves; the first plan after it must show **"No changes"** — that is the durable gate. For the refactor window the apply workflow is temporarily switched to manual dispatch (a `workflow_dispatch` trigger added, the push trigger gated), reverting afterwards. **#282 is explicitly excluded from this window**: hashing trigger entries changes trigger *values* and therefore forces a reinstall — it remains deferred to the next planned reinstall window and is named here only to prevent accidental bundling. Neither state addresses, topology, nor trigger identity change in this step (#268).

4. **Credential topology** (#329): the module repo holds **only** dedicated e2e credentials — its own OVH Public Cloud project, its own object-storage cloud user and state bucket (solving quota isolation, blast radius, per-object ownership, and the 2-key limit in one move), and a **separate e2e tailnet** whose name reveals nothing about production. The module repo's OVH **API** credential is either a separate OVH account or a consumer key whose route grants are restricted to the e2e project's `/cloud/project/{id}/*` paths and explicitly exclude `/dedicated/*` — the project-scoped cloud user isolates object storage and OpenStack only, while an unrestricted account-scoped API key would still reach the production dedicated server's reinstall endpoint. Prod-state-writing workflows exist in exactly one repo at any moment; GitOps enablement flips atomically if that ever moves. The module repo's `e2e` environment requires reviewer approval and a branch policy; paid runs trigger only via `workflow_dispatch` or a maintainer-applied label re-checked in-job with a same-repo guard; **`pull_request_target` is banned**; all identifiers live in environment secrets (#300); e2e plan/apply output ships as redacted artifacts, never PR comments; sensitive module outputs are marked `sensitive = true`.

5. **Release engineering** (#321): semver with a tag ruleset (`v*`: creation restricted, update/delete blocked). Tags stay **signed and manual**; automation produces only the CHANGELOG/version-bump PR. SBOM attaches on release (existing flow). **Breaking change**, stated in consumer terms: any change that forces resource replacement or node reinstall on `tofu apply` of an otherwise-unchanged consumer configuration (the reinstall-trigger-identity case, #268/#282, is the internal name for this), plus variable renames/removals and default flips. Pre-1.0.0, breaking changes bump MINOR per semver 0.x semantics; the strict MAJOR rule takes effect at v1.0.0. **Module repo deliverables** beyond code: CONTRIBUTING.md, SECURITY.md, and CODE_OF_CONDUCT.md adapted for a module audience; terraform-docs-generated input/output/requirements tables in each module README, enforced in CI so registry-rendered docs never drift; and a version-support statement covering which MAJOR lines receive fixes, the supported OpenTofu/Terraform core range, and the supported Talos/Kubernetes compatibility matrix.

6. **Registry: OpenTofu only** (#338). The module ships 0.x through the multi-node shakeout; v1.0.0 and the registry submission (via the `opentofu/registry` issue form; subsequent tags auto-index) happen once the N-node/adapter contract is exercised by the Tier-1 e2e examples. The HashiCorp registry is not used (ADR-0014 stance; publication also requires granting its registry OAuth access to the GitHub account). Terraform users consume via git-tag source — to keep that fallback real, module HCL stays Terraform-compatible (no OpenTofu-exclusive language features) and the module declares its own conservative `required_version` floor rather than inheriting this repo's `>= 1.10.0`, which is a backend-locking concern of the consumer root, not of the module. Terraform support is best-effort and its minimum version is stated in the module README.

7. **Support tiers and test matrix** (policy summary; full text lands as TESTING.md/SUPPORT.md, #327): **Tier 1** — the opinionated default path; the blessed `examples/` configurations are the *entire* e2e matrix, and option interactions are tested only within Tier 1. The blessed examples run on `node-ovh-publiccloud` (hourly-billable, ephemeral); `node-ovh-baremetal` — the production path — is **Tier 2 for e2e purposes**: `tofu validate` + mocked/plan-level `tofu test` per PR, plus periodic manual real-hardware validation, because OVH dedicated servers are monthly-billed and cannot be provisioned per-PR (the #337 spike decides whether hourly Metal Instances can lift it into Tier 1). **Tier 2** generally — flag-gated alternatives (e.g. the second-class fabric arms from planned ADR-0017, storage variants): validate + contract tests + occasional manual runs; never per-PR e2e. The published SUPPORT.md leads with the guarantee and the promotion path: *e2e coverage is guaranteed for the configurations in `examples/`; other combinations are contract-tested and community-supported; the way into the Tier 1 matrix is contributing your configuration as a blessed example.*

8. **Renovate split**: the module repo keeps the `variables.tf` custom managers and provider groups; the consumer **exact-pins** the module version (`= x.y.z`, never `~>` ranges) and advances it only via a reviewed Renovate bump PR. `talos_version`/`kubernetes_version` remain Renovate-ignored in both repos (support-matrix lockstep, the #269 lesson), encoded as explicit ignore rules so a preset refresh cannot resurrect them.

9. **ADR-0014 amendment**: CI becomes two-repo (module CI + consumer GitOps), while the production workspace stays single. Native S3 locking (`use_lockfile`) is load-bearing because production-state writers share no GitHub Actions concurrency domain — consumer CI applies, the operator's local upgrade-mode applies, and (transitionally, if ever) a second repo — not merely because concurrency groups don't span repositories. The one-writer-repo rule in decision 4 keeps the *repo* dimension of that set fixed at one.

## Consequences

**Positive:**

- A community-reusable module in a niche where the prior-art survey (2026-07-08) found no existing equivalent — provider-agnostic Talos core, and the only Talos-on-OVH-dedicated implementation — with the upgrade lifecycle as its differentiating feature (e2e-verified per-PR on the Public Cloud adapter; validated on dedicated hardware by periodic manual runs).
- This repo's operational hardening is untouched: zero state migration, zero secret re-provisioning, no two-writer-repo window.
- e2e blast radius fully isolated from production at every layer: cloud project, API credential scope, object-storage identity, state bucket, tailnet, GitHub environment.
- The zero-diff gate converts the riskiest step (mass state-address rewrite on the only production node) into a mechanically verifiable no-op.

**Negative / accepted risks:**

- **Two repos to maintain**: duplicated CI plumbing, split Renovate configs, and a grown secret-sync surface — the consumer's tfvars-as-secret must track module input changes. Mitigation: a consumer CI step validates materialized tfvars on module-bump PRs so input renames fail in the PR, not at apply on main; shrinking the opaque tfvars secret to individual `TF_VAR_*` values remains an option.
- **Module repo enters the consumer's trust base.** The consumer executes provider config, provisioners, and inline manifests fetched from `terraform-talos-cluster`; a compromised tag is a supply-chain path into production. Mitigations: exact-version pinning advanced only by a reviewed bump PR, immutable signed tags (decision 5's ruleset blocks update/delete), and the consumer's plan gate on every bump as the review checkpoint. Residual risk — a subtle malicious change surviving plan review — accepted.
- **The differentiating path is the least e2e-tested.** Per-PR e2e covers the Public Cloud adapter; the dedicated-server adapter relies on contract tests plus periodic manual runs (decision 7). Accepted until #337 resolves whether hourly bare-metal e2e is possible at all.
- **OVH API scoping is manual.** OVH has no native per-project API-credential scoping, so the e2e credential's route restriction (decision 4) is deliberate configuration that must be re-verified when rotated.
- **Release latency on prod changes**: module change → tag → consumer bump PR → review. Accepted; local `source` override is the documented development loop.
- **Paid, babysat e2e**: OVH offers budget email alerts but no hard spend cap, so the pipeline itself is the kill switch (serialized concurrency, destroy-in-`always()`, TTL sweeper, billing poll — #331). A cancelled runner can strand resources until the sweeper passes.
- **Single-maintainer concentration.** Manual signed-tag releases, babysat paid e2e, and two-repo CI all depend on one operator; there is no failover for a stranded e2e sweep or an urgent release. Mitigation: the release and sweeper runbooks (#327/#331) are written to be executable by a second operator; onboarding one is the standing exit from this risk.
- The extraction is deliberately slower than a naive move — the gate is the point, not overhead.
